# Algolia API Key Exposure: Priority Assessment & Plan

**Status:** Draft 
**Owner:** TBD
**Date:** 2026-05-18

## TL;DR

- **Issue.** The Algolia search API key bundled into our three Enterprise frontends is unscoped — anyone holding it can query the catalog data of every enterprise customer in the application, not just their own.
- **Severity.** Medium (CVSS 5.3). No write access, no PII, no credentials at risk. The real exposure is cross-tenant data isolation: an attacker can map which enterprise customers curate which content.
- **Fix.** Stop bundling the public key. Switch all three frontends to per-request "secured" keys generated by `enterprise-catalog`. Two of the three apps already do this for the happy path; we're removing fallbacks and building a new endpoint for the unauthenticated public-catalog app.
- **Rollout.** Four phases (admin + learner portal cleanup → new public endpoint → shared package update → key rotation). Timeline TBD.
- **Decisions needed from product.** When (if ever) to restrict the public-catalog site from showing `video` record types. See open questions.

## Background

A security researcher flagged that the Algolia public/search API key is bundled into the JavaScript of our three Enterprise frontends:

- `frontend-app-admin-portal` — authenticated; uses secured key but **falls back to the public key** on failure
- `frontend-app-learner-portal-enterprise` — authenticated; same fallback behavior (secured key delivered via BFF)
- `frontend-app-enterprise-public-catalog` — unauthenticated SPA; **uses the public key directly, no secured-key path exists**

The backend already has a working secured-key endpoint at `GET /api/v1/enterprise-customer/<uuid>/secured-algolia-api-key/` which scopes results by `enterprise_catalog_query_uuids`, `userToken`, `validUntil` (1h), and `restrictIndices`.

## Severity: Medium (CVSS 5.3)

**Plain-English risk.** An attacker can read catalog content and enumerate which enterprise customers have curated what. They cannot modify any data, cannot access user information, and cannot impact availability.

**CVSS v3.1 vector:** `AV:N/AC:L/PR:N/UI:N/S:U/C:L/I:N/A:N` → **Base score 5.3 (Medium)**

| Metric | Value | Rationale |
|---|---|---|
| Attack Vector | Network | Key is in the bundled JS; pulled with a single HTTP request. |
| Attack Complexity | Low | No timing, race, or configuration prerequisites. |
| Privileges Required | None | Bundled JS is unauthenticated. |
| User Interaction | None | Attacker uses the key directly against Algolia. |
| Scope | Unchanged | Impact confined to Algolia search context. |
| Confidentiality | Low | Catalog metadata, mostly already public on edx.org; the leak is cross-tenant *curation mapping* (which enterprise has which courses) plus the ability to enumerate enterprise/catalog UUIDs as facets. No PII, no credentials. |
| Integrity | None | Key has search-only ACL; no writes possible. |
| Availability | None | Algolia rate-limits at the key level; no DoS on our systems. |

Why not Critical: no write/admin scope, no PII, no credentials, content largely public elsewhere. Why not Low: cross-tenant data isolation is a customer-trust expectation in B2B, and the key allows enumeration of every enterprise/catalog UUID via facet queries.

Algolia is explicit that search keys are designed to be public ([docs](https://www.algolia.com/doc/guides/security/api-keys#main-and-secured-api-keys)); the issue is that ours is unscoped (no `filters`) and bundled into apps where a per-tenant secured key is the appropriate trust boundary.

## Plan

### Phase 1 — Authenticated frontends (fastest, biggest reduction)

1. Remove the public-key fallback in `frontend-app-admin-portal` (`src/components/algolia-search/useAlgoliaSearch.ts:111-123`) and `frontend-app-learner-portal-enterprise` (`src/components/app/data/hooks/useAlgoliaSearch.ts:182-196`). Fail closed on secured-key fetch errors.
2. **Failure UX — reuse existing prior art.** Learner portal already has `src/components/search-unavailable-alert/SearchUnavailableAlert.tsx` (Paragon `Alert variant="danger"` with refresh/contact-support guidance) — render it when the secured-key fetch errors out. Admin portal uses `react-query` with `defaultQueryClientRetryHandler` (`src/utils.js:524`) for transparent retry; surface a `Paragon Alert` (pattern: `src/components/CodeModal/ModalError.jsx`, `src/components/settings/SettingsApiCredentialsTab/FailedAlert.jsx`) once retries are exhausted.
3. Stop publishing `ALGOLIA_SEARCH_API_KEY` to the build for these two apps. Remove from `.env*` and deployment configs.
4. Add client-side refresh of the secured key well before `validUntil` so users mid-session don't hit the failure path.

### Phase 2 — Public catalog (new endpoint required)

1. Add an **unauthenticated** endpoint, e.g. `GET /api/v1/public/algolia-api-key/`, that returns a secured key restricted to an allow-list of "publicly visible" catalog queries. Today's visible set is the queries whose `title` matches `EDX_FOR_SUBSCRIPTION_TITLE` (default `Subscription`) and `EDX_ENTERPRISE_ALACARTE_TITLE` (default `A la carte`). The public-catalog frontend already filters records by those titles via `enterprise_catalog_query_titles` (see `src/components/courseCard/CourseCard.jsx:51,58`, `src/components/programCard/ProgramCard.jsx:26,29`, `src/components/catalogSearchResults/associatedComponents/catalogBadges/CatalogBadges.jsx:13,20`, `src/utils/catalogUtils.ts:8`).
2. The backend resolves those titles to `CatalogQuery.uuid`s at request time and applies a `filters` restriction of `enterprise_catalog_query_uuids:<uuid> OR ...` — UUIDs are stable, titles are admin-editable, so we trust the UUID. The allow-list of titles lives in Django settings.
3. Use `attributesToRetrieve` in the secured-key restrictions to strip fields the public UI doesn't need (e.g., `enterprise_catalog_uuids`, `enterprise_customer_uuids`, `academy_uuids`, internal scoring fields). This is the right knob for field-level scoping; `restrictSources` is for IP ranges and isn't useful here. No `userToken`; short `validUntil`; `restrictIndices` set.
4. Throttle with DRF `AnonRateThrottle` (custom scope, e.g. `algolia_public_key`): **10/minute and 100/hour per IP**. The public catalog gets ~1,000 views/week (~6/hour peak), and each session refetches at most once per `validUntil` window — both limits sit comfortably above real traffic and immediately cut off scrapers. No CDN caching — the client holds the key until `validUntil` and refreshes on its own; caching at the edge is unnecessary and complicates rotation.
5. Update `frontend-app-enterprise-public-catalog` (`src/components/catalogs/data/hooks.jsx:9-18`) to fetch from this endpoint instead of reading `config.ALGOLIA_SEARCH_API_KEY`. Remove the env var.

### Phase 3 — Shared package

Update `@2uinc/frontend-enterprise-catalog-search` so its `SearchContext` accepts a secured-key resolver (sync or async) and removes any default that reads a public key from config. Coordinate with `edx/frontend-enterprise`.

### Phase 4 — Rotate & verify

Once all three frontends are off the public key for at least one deploy cycle, **rotate the existing `ALGOLIA.SEARCH_API_KEY`** in stage then prod. Verify no app breaks. This invalidates any cached copies of the leaked key.

## Index strategy

Recommend **keeping a single index** and relying on filtered secured keys rather than maintaining a separate public-only index. Reasons: less indexing/ops surface; the secured-key filter is the same trust boundary either way; we'd otherwise need to duplicate the indexing pipeline. Revisit if the public allow-list grows complex.

## Public-catalog secured-key restrictions (recommended)

Apply all of the following as restrictions when generating the secured key for `GET /api/v1/public/algolia-api-key/`:

**`filters`** — combine these clauses:

1. Catalog-query scope (required): `enterprise_catalog_query_uuids:<uuid1> OR enterprise_catalog_query_uuids:<uuid2>` (resolved from the `EDX_FOR_SUBSCRIPTION_TITLE` / `EDX_ENTERPRISE_ALACARTE_TITLE` allow-list).
2. Content-type scope (required): start at `(content_type:course OR content_type:program OR content_type:video)` — matches the current state of the public catalog. The backend also indexes `courserun` and `learnerpathway` (`enterprise_catalog/apps/catalog/constants.py:14-18`), which the public UI was never designed to surface; exclude those.
3. Video toggle (configurable): a `PUBLIC_CATALOG_ALLOW_VIDEOS` setting controls whether `content_type:video` is included in clause 2. **Default `true`** to preserve today's behavior; product can flip it to `false` at any time without an indexing change.

**Other restrictions:** `validUntil` (short window, e.g. 1h), `restrictIndices` (primary + replica), no `userToken`.

**`attributesToRetrieve`** — explicit allow-list, derived from `ALGOLIA_FIELDS` in `enterprise_catalog/apps/catalog/algolia_utils.py:72-144`:

| Category | Allow | Deny | Reason |
|---|---|---|---|
| Identifiers | `objectID`, `uuid`, `key`, `aggregation_key`, `course_keys`, `course_run_key` | `enterprise_catalog_uuids`, `enterprise_customer_uuids`, `academy_uuids`, `enterprise_catalog_query_uuids`, `academy_tags` | Internal tenancy and curation IDs; first three are already `unretrievableAttributes` but belt-and-suspenders. `academy_tags` is internal taxonomy. |
| Discriminators | `content_type`, `course_type`, `course_length`, `learning_type`, `learning_type_v2`, `program_type`, `type`, `availability` | — | Needed for filtering/rendering. |
| Display content | `title`, `subtitle`, `short_description`, `full_description`, `additional_information`, `outcome`, `prerequisites`, `prerequisites_raw`, `language`, `metadata_language`, `level_type`, `subjects`, `skill_names`, `skills`, `programs`, `program_titles`, `partners`, `partner`, `authoring_organizations`, `org`, `learning_items`, `course_details`, `additional_metadata`, `normalized_metadata`, `entitlements`, `duration`, `is_new_content` | — | Already-public marketing data. |
| Images / URLs | `image_url`, `original_image_url`, `card_image_url`, `banner_image_url`, `logo_image_urls`, `marketing_url` | — | Already-public marketing assets. |
| Pricing & runs | `prices`, `first_enrollable_paid_seat_price`, `advertised_course_run`, `course_runs`, `upcoming_course_runs` | — | Same prices appear on edx.org marketing pages. |
| Ratings | `avg_course_rating`, `reviews_count` | `recent_enrollment_count`, `course_bayesian_average` | Display ratings are public on edx.org. Enrollment count and Bayesian average are useful for ranking but reveal usage signal — leave them in the *ranking formula* (the secured key doesn't unset that), just don't return them. |
| Video subfields | `transcript_languages`, `video_ids`, `video_usage_key`, `video_skills`, `transcript_summary` | — | Needed while `PUBLIC_CATALOG_ALLOW_VIDEOS=true` (default). If the toggle is flipped off, video records won't be returned anyway and these fields become moot — no allow-list change needed at that point. |
| Internal | — | `visible_via_association`, `enterprise_catalog_query_titles`, `created` | See **Badge decision** below for `enterprise_catalog_query_titles`. `visible_via_association` is an internal flag; `created` is the indexing timestamp. |

### Badge decision: `enterprise_catalog_query_titles`

The public-catalog UI uses this field at three call sites to render "Subscription" / "À la carte" badges on result cards:

- `frontend-app-enterprise-public-catalog/src/utils/catalogUtils.ts:6-14` — `checkSubscriptions(courseAssociatedCatalogs)` checks for `process.env.EDX_FOR_SUBSCRIPTION_TITLE`.
- `frontend-app-enterprise-public-catalog/src/components/catalogSearchResults/associatedComponents/catalogBadges/CatalogBadges.jsx:12,19` — renders both badges if the record's `enterprise_catalog_query_titles` array contains either env-var title.
- `frontend-app-enterprise-public-catalog/src/components/courseCard/CourseCard.jsx:50,57` — same pattern inline on course cards (`ProgramCard.jsx:26,29` repeats it for programs).

All three boil down to: `record.enterprise_catalog_query_titles.includes(<env title>)`.

The field is a **leak** if exposed as-is. A single content record is tagged with the title of *every* `CatalogQuery` that selects it, across every enterprise — so anyone holding the public secured key could page through results and reconstruct the cross-tenant curation map (which enterprise has named their catalog query what). That's exactly the tenancy concern driving this work.

**Options:**

1. **Keep the field allow-listed.** Cheapest. Accepts the leak.
2. **Drop the field; rework badges off active facet refinement.** Lower badge fidelity — a course in *both* Subscription and À la carte can only render one badge at a time (the active one), and on the "all results" landing view you can render neither without another signal. Cheap on the backend, modestly invasive on the frontend.
3. **Index a derived `public_catalog_query_titles` field** that contains only the intersection of `enterprise_catalog_query_titles` with the public allow-list. Expose that field instead. Frontend rename only; no behavior change. One-time addition in `enterprise_catalog/apps/catalog/algolia_utils.py` (`ALGOLIA_FIELDS` + the serializer/transform that builds the record).

**Recommendation: option 3.** Preserves UI behavior, closes the leak cleanly, and the indexing change is small and isolated. Option 2 is the fallback if indexing-change appetite is low.
