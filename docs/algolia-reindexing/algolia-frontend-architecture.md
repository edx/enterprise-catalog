# Algolia Frontend Architecture

## Index Name Sources

| Component | Index Name Source | API Key Source |
|-----------|------------------|----------------|
| **enterprise-catalog** (writes) | `settings.ALGOLIA['INDEX_NAME']` | `settings.ALGOLIA['API_KEY']` (admin key) |
| **enterprise-access** (BFF) | Does not use index name | Passes through secured key from enterprise-catalog |
| **frontend-app-learner-portal-enterprise** | `env..ALGOLIA_INDEX_NAME` (env var) | Secured key from BFF, fallback to `ALGOLIA_SEARCH_API_KEY` env |
| **frontend-app-admin-portal** | `env.ALGOLIA_INDEX_NAME` (env var) | Secured key from enterprise-catalog, fallback to `ALGOLIA_SEARCH_API_KEY` env |
| **frontend-app-enterprise-public-catalog** | `env.ALGOLIA_INDEX_NAME` (env var) | `ALGOLIA_SEARCH_API_KEY` (env var, no secured key) |

## Key Insight: Secured API Key Restrictions

The secured API key generated by enterprise-catalog includes `restrictIndices`:

```python
restrictions = {
    'filters': catalog_query_filter,
    'validUntil': valid_until_unix,
    'userToken': user_id,
    'restrictIndices': [
        settings.ALGOLIA['INDEX_NAME'],
        settings.ALGOLIA['REPLICA_INDEX_NAME'],
    ],
}
```

This means frontends can **only search indices listed in `restrictIndices`**. If a frontend's `ALGOLIA_INDEX_NAME` doesn't match what's in the secured key, searches will fail.

## Cutover Implications

For zero-downtime cutover, we need to:
1. Ensure the secured key allows both old and new index names during transition
2. Update frontend configs (can be gradual)
3. Once all frontends migrated, remove old index from `restrictIndices`
